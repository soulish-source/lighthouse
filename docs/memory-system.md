# Memory System Design

## 为什么 Lighthouse 必须重视记忆

长篇网文的核心问题不是“这一章怎么写”，而是：

- 之前写过什么会不会忘
- 人物状态有没有持续继承
- 伏笔有没有在正确时机回收
- 新增设定会不会打脸旧设定
- 后续章节能不能准确取回真正相关的历史事实

所以 Lighthouse 的记忆系统不应该只是聊天历史，而应该是 **结构化事实 + 摘要索引 + 可回写的状态层**。

---

## 记忆分层

### 1. 结构化事实层

保存适合机器处理和校验的事实。

例如：

- 人物身份、阵营、能力、伤势、关系变化
- 世界规则、地点规则、物品能力
- 伏笔埋设 / 强化 / 回收
- 时间推进和关键事件

用途：

- 查询
- 过滤
- 一致性检查
- 时间线排序
- 章节上下文拼装

### 2. 章节摘要层

为每章生成低成本、可召回的摘要对象，例如：

- 这一章做了什么
- 哪些人物状态发生变化
- 哪些伏笔被触发
- 哪些设定被新增或修正
- 这一章对后续最重要的约束是什么

用途：

- 提供给创作 Agent 做快速回顾
- 作为向量召回素材
- 给编辑/作者做浏览式理解

### 3. 工作记忆层

面向当前章节任务的临时上下文。

可能包含：

- 当前章节提纲节点
- 上一章定稿摘要
- 相关人物状态
- 必须命中的伏笔
- 当前时间点和地点约束
- 模型配置和写作策略

用途：

- 控制单次创作任务的上下文窗口
- 减少全量历史灌入导致的噪声

---

## 写后回写流程

建议链路：

1. 章节定稿
2. 触发记忆抽取任务
3. 生成结构化事实变更集
4. 生成章节摘要
5. 更新人物动态状态 / 伏笔状态 / 时间线事件
6. 产出差异报告，必要时人工确认后再落库

回写对象示例：

- `character_state_changes`
- `memory_facts`
- `foreshadow_updates`
- `timeline_events`
- `chapter_summary`

---

## 写前检索流程

当准备生成下一章时，不应该直接把全书喂给模型，而应先构建相关上下文。

推荐步骤：

1. 根据章节提纲定位目标
2. 查询直接依赖章节
3. 召回关联人物最新状态
4. 召回关联伏笔与预期回收节点
5. 召回相关时间线事件
6. 补充语义相近的章节摘要 / 设定片段
7. 将这些内容压缩成任务工作记忆

这样做的好处：

- 相关性更高
- token 更可控
- 更容易发现缺失数据
- 更适合做一致性检查

---

## 检索策略

### 精确检索

适用于：

- 按人物 ID 找最新状态
- 按章节找直接上下文
- 按伏笔 ID 找当前状态
- 按故事日历找特定时间事件

### 语义检索

适用于：

- 找与当前冲突主题相似的历史场景
- 找类似人物情绪转折
- 找可能相关但未显式关联的设定片段

### 混合检索

最适合 Lighthouse：

- 先用结构化条件缩小范围
- 再用语义召回补边角信息
- 最后做人工或规则重排

---

## 一致性审查如何使用记忆

审查阶段建议至少读取：

- 当前章节草稿
- 直接前序章节摘要
- 相关人物最新状态
- 相关世界规则
- 相关伏笔状态
- 当前时间点前后的关键事件

审查输出应指向可执行问题：

- 哪句话疑似违背历史设定
- 证据来自哪一章 / 哪条记忆
- 是硬冲突、软风险还是待确认项

---

## 存储建议

### 关系型数据库

适合存：

- 结构化事实
- 人物状态
- 伏笔状态
- 时间线事件
- 审查结果

### 向量库

适合存：

- 章节摘要
- 设定片段
- 关系变化摘要
- 历史冲突案例

### 对象存储

适合存：

- 章节全文
- 审查报告全文
- prompt / response 快照

---

## 当前阶段落地建议

第一阶段先不要把记忆系统做得过度复杂，先把接口和对象想清楚：

- 前端用页面结构表达“记忆是系统核心部分”
- 后端 DTO 先给出角色、伏笔、时间线、任务、模型配置这些可扩展对象
- 文档明确后续回写与检索的真实工作流

## 下一步最值钱的实现

1. 定义 `memory_facts` 与 `chapter_summary` 数据结构
2. 给章节定稿增加一个“回写预览”接口
3. 让一致性审查结果直接引用历史证据
4. 在前端章节页展示“将被召回的上下文包”
