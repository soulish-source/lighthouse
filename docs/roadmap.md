# Roadmap

## Phase 1：产品方向可展示（当前阶段）

目标：让 Lighthouse 不再只是“长篇写作 agent”的一句话，而是一个能看出产品骨架的工程原型。

交付重点：

- 后端领域模型与基础 API 结构
- 前端多页面导航与核心工作区壳子
- 长篇创作相关文档：领域、记忆、路线图
- 用 mock 数据明确产品对象，而不是停留在 landing page

完成标准：

- 能看到项目概览、提纲、章节、角色设定、伏笔时间线、模型配置
- 后端接口至少能返回统一结构的样例数据
- README 和 docs 足够让人理解下一步怎么做

---

## Phase 2：最小闭环可运行

目标：打通“项目 → 提纲 → 章节 → 审查”的最小真实链路。

建议范围：

- PostgreSQL 落库
- 项目、提纲、章节 CRUD
- 章节版本与审查结果存储
- 前端从 mock 切到真实 API
- 基础表单、详情页和状态流转

完成标准：

- 可以创建一个作品、补提纲、管理章节卡片
- 可以发起章节草稿任务并保存结果
- 可以看到基础审查结果和时间线条目

---

## Phase 3：记忆系统与一致性引擎

目标：建立 Lighthouse 与普通写作工具的真正差异化。

建议范围：

- 章节定稿后的结构化记忆回写
- 人物动态状态、关系变化、设定新增抽取
- 伏笔状态更新与超期提醒
- 时间线冲突检测
- 一致性审查结果标准化

完成标准：

- 任一章节可以关联“写前上下文 / 写后回写 / 审查结果”
- 系统能明确指出人物 OOC、时间错误或伏笔遗忘风险

---

## Phase 4：Agent 编排与模型治理

目标：让创作、抽取、审查成为稳定工作流，而非手动拼接 prompt。

建议范围：

- 任务队列 / 重试 / 状态机
- 模型配置中心与 provider 抽象
- prompt preset 与策略版本化
- 成本、限流、失败兜底
- 长任务审计与可观测性

完成标准：

- 同一章节任务可以被不同 agent/profile 执行
- 模型切换不影响前端和主流程

---

## Phase 5：协作与产品化

目标：从个人创作工作台进化到真正可用产品。

建议范围：

- 登录、权限、团队协作
- 评论、批注、审校流
- 作品分支与平行剧情实验
- 导出、版本比较、变更追踪
- 部署、CI/CD、监控告警

完成标准：

- 具备基础产品运营形态
- 支持多人协作和长期维护

---

## 当前判断

最该避免的坑：

1. 一开始就重投入在模型接入，结果基础资产结构不稳
2. 把长篇系统做成聊天页，丢掉结构化创作管理的核心价值
3. 过早引入复杂 infra，却没有先把领域对象和交互面板说清楚
