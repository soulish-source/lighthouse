# Lighthouse 架构设计

## 1. 项目定位

Lighthouse 是一个面向**长篇网文创作**的 agent 系统。它不是只生成单章文本的“写作机器人”，而是一个围绕长线叙事构建的创作基础设施，目标是帮助作者持续维护作品的结构稳定性与设定一致性。

系统重点支持以下核心对象：

- **作品 / 书籍**：整部小说的顶层容器
- **卷 / 篇章 / 故事阶段**：中层结构拆分
- **提纲**：全书提纲、分卷提纲、章节提纲
- **章节**：章节草稿、版本、审校记录
- **人物 / 组织 / 地点 / 物品 / 能力 / 世界规则**：设定资产
- **伏笔**：埋点、预期回收位置、回收状态
- **时间线**：剧情事件在故事内时间轴上的排序与约束
- **记忆**：从已完成章节中抽取的稳定事实与动态状态
- **一致性检查结果**：冲突、缺失、风险提示

---

## 2. 核心设计目标

### 2.1 支持长线创作，而非一次性生成

长篇网文创作的难点不在于生成一段文字，而在于：

- 前后设定是否一致
- 人物动机是否连贯
- 伏笔有没有遗忘
- 时间线是否自洽
- 已发生的剧情是否被后续章节正确继承

因此 Lighthouse 采用“**结构化创作资产 + agent 工作流**”的方式，而不是仅保留自然语言上下文。

### 2.2 把“记忆”作为一等公民

每一章写完后，系统不只保存正文，还要把关键信息回写到长期记忆层，包括：

- 人物状态变化
- 关系变化
- 新增设定
- 世界规则补充
- 伏笔埋设或回收
- 时间推进
- 已确认事实

### 2.3 一致性检查优先于无约束生成

章节生成前、生成中、生成后都要有一致性检查：

- 生成前：检查提纲、人物状态、时间线是否齐备
- 生成中：提供设定约束与关键事实
- 生成后：审查是否出现逻辑冲突、设定漂移、时间错误、人物 OOC

## 4. 核心业务模块

### 4.1 提纲系统（Outline）

提纲系统负责从“大纲 → 分卷 → 章节计划”逐层拆解。

建议支持：

- 全书主线与副线
- 分卷目标、冲突、高潮、结尾
- 章节目标、视角、冲突点、信息揭示点
- 每章与前后章节的依赖关系

作用：

- 为章节生成提供明确边界
- 让 agent 在写作时遵循结构目标
- 为伏笔与时间线提供锚点

### 4.2 章节系统（Chapter）

章节是创作的核心生产对象，至少包含：

- 所属作品/卷/提纲节点
- 章节标题
- 章节目标
- 草稿正文
- 版本历史
- 审核状态
- 关联人物/设定/伏笔/时间点

章节工作流建议：

1. 选定提纲节点
2. 拉取相关记忆与设定
3. 生成章节草稿
4. 执行一致性检查
5. 人工修订
6. 回写记忆与状态

### 4.3 人物与设定系统（Character / Lore）

这是长篇创作最容易失控的部分。

应管理：

- 人物基础信息：姓名、年龄、阵营、身份
- 人物长期属性：性格、口头禅、能力、弱点、目标
- 人物动态状态：伤势、关系、立场、已知信息
- 世界设定：修炼体系、政治结构、地理、规则边界
- 物品/组织/地点等实体

关键点：

- 将“静态设定”与“动态状态”拆分
- 允许标注事实来源章节
- 允许标注可信度与生效范围

### 4.4 伏笔系统（Foreshadowing）

伏笔系统不是简单备注，而是“可追踪的叙事承诺”。

每个伏笔建议包含：

- 伏笔内容
- 埋设章节
- 涉及人物/事件/物品
- 预期回收范围（某卷/某章/长期）
- 当前状态：已埋设 / 待回收 / 已回收 / 作废
- 风险等级：高风险遗忘 / 可延后

系统可提供：

- 临近回收节点提醒
- 已超期未回收伏笔告警
- 回收后自动关联章节

### 4.5 时间线系统（Timeline）

长篇小说最常见问题之一是时间错乱。

时间线系统应至少处理：

- 故事内绝对时间（如某年某月某日）
- 相对时间（第 3 天、三个月后）
- 事件发生顺序
- 人物在某时间点的状态
- 地点迁移与旅行耗时

一致性检查示例：

- 同一人物是否在两个地点同时出现
- 角色伤势恢复是否超出设定时间
- 某事件是否发生在因果前提之前
- 伏笔回收是否早于埋设

### 4.6 记忆回写系统（Memory Write-back）

这是 Lighthouse 的关键差异点。

每次章节定稿后，从文本中抽取结构化记忆并回写：

- 新事实：人物身份暴露、地点新增规则、法宝能力说明
- 状态变化：关系破裂、晋级、受伤、阵营改变
- 剧情进展：主线推进、副线埋点
- 伏笔变更：埋设 / 强化 / 回收
- 时间推进：故事日历更新

记忆回写建议采用两层：

1. **结构化层**：供查询、过滤、审查、计算
2. **摘要层**：供 agent 在生成时快速读取

### 4.7 一致性检查系统（Consistency Check）

一致性检查是整个系统的质量守门员。

至少覆盖：

- **人物一致性**：性格、立场、知识边界、能力边界
- **设定一致性**：世界规则、力量体系、组织关系
- **剧情一致性**：因果链、动机、信息揭示顺序
- **伏笔一致性**：是否遗漏、是否误回收、是否冲突
- **时间一致性**：顺序、时长、伤势恢复、路程时间
- **文风/视角一致性**：POV 是否漂移，叙述口吻是否偏离

检查结果应输出为可执行建议，而不只是报错：

- 冲突描述
- 证据来源章节
- 严重等级
- 建议修复方式

## 5. Agent 工作流建议

### 5.1 提纲生成 Agent

输入：作品定位、题材、世界观、核心冲突
输出：全书提纲、分卷计划、章节草案

### 5.2 章节创作 Agent

输入：章节提纲 + 相关记忆 + 人物状态 + 时间线约束
输出：章节草稿

### 5.3 记忆抽取 Agent

输入：定稿章节
输出：新增事实、状态变化、伏笔变化、时间推进

### 5.4 一致性审查 Agent

输入：章节草稿 + 历史记忆 + 设定库 + 时间线
输出：冲突报告、修复建议

### 5.5 编排器（Orchestrator）

典型链路：

1. 选择章节任务
2. 拉取相关提纲与上下文
3. 加载人物/设定/时间线/伏笔记忆
4. 调用章节创作 agent
5. 调用一致性审查 agent
6. 人工确认
7. 回写记忆
8. 更新伏笔与时间线状态

## 6. 前后端职责划分

### Frontend

负责：

- 作品总览 Dashboard
- 提纲编辑器
- 章节创作工作台
- 人物/设定管理页面
- 伏笔看板
- 时间线视图
- 一致性报告与修复建议展示

### Backend

负责：

- 领域模型与业务规则
- agent 工作流编排
- 记忆抽取与回写
- 一致性检查逻辑
- 数据存储与检索
- API 与鉴权

---

## 7. 未来演进方向

- 接入 PostgreSQL 保存结构化创作资产
- 接入向量检索支持“按语义召回相关章节/设定”
- 引入事件溯源记录人物与世界状态变化
- 支持多模型协作：创作、审查、总结、抽取分工
- 为编辑/作者/审校提供多角色工作流
- 支持作品版本分叉与平行剧情试验

---

## 8. 当前阶段结论

当前仓库初始化的重点不是一次性把系统做完，而是先落下一个**稳健、可扩展、可表达产品方向**的前后端分离骨架。

短期优先级建议：

1. 定义核心领域模型（作品、提纲、章节、人物、伏笔、时间线）
2. 打通最小闭环：创建作品 → 创建提纲 → 创建章节 → 章节审查
3. 预留记忆回写与一致性检查接口
4. 逐步引入数据库、LLM 编排与测试体系
